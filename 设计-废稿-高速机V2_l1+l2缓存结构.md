### 二、精准量化计算（完全按你给的行程数据）
固定你给出的核心行程参数：
- 单次取料往返总行程：`S往返=30cm`
- 原方案单颗拼豆总行程：`S原单颗=30cm`（每颗都跑完整往返）
- 新方案单颗落料平均行程（作业区内同色点间移动）：`S新单颗=5cm`

#### 1. 原方案（无缓存）耗时&产能计算
**单颗拼豆完整周期耗时**：
```
T原单颗 = 往返行程耗时 + 单颗取料耗时 + 单颗落料耗时
= (30cm ÷ 15cm/s) + 0.2s + 0.2s
= 2s + 0.4s = 2.4s/颗
```
**16颗拼豆总耗时**：`T原16颗=16×2.4s=38.4s`
**小时理论产能**：`3600s ÷ 2.4s/颗=1500颗/小时`

#### 2. 新方案（L1+L2缓存）耗时&产能计算
以**16颗同色拼豆为一个批量单元**（L1缓存100%利用率，同色连续图案场景）计算：
**16颗拼豆总耗时拆解**：
1.  补料往返行程耗时：`30cm ÷ 15cm/s=2s`
2.  L1满管补料辅助耗时：`0.5s`
3.  16颗落料的移动总耗时：`(16×5cm) ÷ 15cm/s≈5.33s`
4.  16颗落料的辅助总耗时：`16×0.2s=3.2s`

**16颗拼豆总耗时**：
```
T新16颗=2s+0.5s+5.33s+3.2s=11.03s
```
**单颗平均耗时**：`T新单颗=11.03s ÷ 16≈0.69s/颗`
**小时理论产能**：`3600s ÷ 0.69s/颗≈5217颗/小时`

- 效率提升倍数：`2.4s÷1.16s≈2.07倍`，哪怕缓存只用1/4，效率依然翻倍。

你说的8根同色L2料管，和L1的运动效率提升是**乘法叠加**，核心解决「人工补料导致的停机损耗」：

// 拼豆机主控制循环
void PixelBean_MainLoop() {
  // 预处理：解析图案文件，按颜色分组，同色坐标按最短路径排序
  Pattern_t targetPattern = ParsePatternFile("custom_bean.pattern");
  ColorGroup_t colorGroups[] = SortByColorAndNearestPath(targetPattern);

  // 主作业循环（按颜色分组作业，减少换色次数）
  for (uint8_t colorIdx = 0; colorIdx < colorGroups.length; colorIdx++) {
    ColorGroup_t currentColor = colorGroups[colorIdx];
    // 同色点批量作业，以L1缓存容量16为一个单元
    for (uint16_t pointIdx = 0; pointIdx < currentColor.pointNum; pointIdx += 16) {
      // 阶段1：L1空管，移动到对应颜色的取料位补满缓存
      MoveTo(GetFeedPosition(currentColor.colorID));
      FeedL1Cache(16); // 批量补满16颗，仅1次动作
      MoveTo(currentColor.points[pointIdx]); // 回到作业区首个落料位

      // 阶段2：连续落料，全程在作业区内移动，无需返回取料位
      uint16_t batchEnd = min(pointIdx + 16, currentColor.pointNum);
      for (uint16_t k = pointIdx; k < batchEnd; k++) {
        MoveTo(currentColor.points[k]); // 同色点间短距离移动，平均5cm
        DropSingleBean(); // 单颗落料，无需额外移动
      }
    }
  }
}
